# 喵选（Meowpick）待对接后端接口需求文档

本文档面向**后端开发团队**，整理了前端“提议（Proposal）系统”目前所需的接口需求。不需要关注前端的具体实现，重点在于数据结构的返回与状态的持久化支持。

## 1. 核心需求背景：引入“提议（Proposal）”概念
在现有的基于“吐槽/评价（Comment）”体系之外，当前版本新增了另一套并行的核心流：“建课提议流程”。
当用户在小程序中无法搜到某门课程时，他们会发起一个“提议”，其他用户也可以对提议进行“附议与投票”。

为了支持前端界面的展示和交互，现梳理出需要新增和调整的后端服务：

## 2. 接口待开发与修改需求清单

### 需求一：全局列表/搜索接口的数据混合能力
**适用场景**：用户在首页搜索，或查看“我的发布”时。
**具体要求**：
*   **兼容混合列表**：列表或搜索接口返回的数据集合中，需要能够同时包含“已存在课程的吐槽”和“待批准的建课提议”。
*   **增加 Type 标识**：在现有的返回实体中新增一个必填字段（如 `item_type` 或 `type`），枚举值为 `comment` 和 `proposal`，前端需要用极其依赖该字段进行样式渲染和点击路由分发。
*   **提议卡片专属数据**：如果某条数据是类型是提议，后端在返回的实体中需要额外包含：
    *   `voteCount` / `likes`（当前的附议/投票总人数）。
    *   `reason` / `content`（发起人编写的求建课理由）。

### 需求二：提议系统专属接口（CRUD）
针对“提议”实体的独立操作，需建立区别于“吐槽”的路由模块组：

1.  **发起提议 (Create Proposal)**
    *   接收课程名称、授课教师、学院等元数据。
    *   接收用户的“提议理由”。
    *   初始化该提议的进度状态记录。

2.  **获取提议详情 (Get Proposal Detail)**
    *   接收提议 ID，返回完整的提议详情。
    *   返回当前登录用户是否已经对该提议进行过“附议”。

3.  **附议/投票操作 (Vote Proposal)**
    *   用户点击附议按钮后调用。
    *   限制：同一个用户对同一个提议只能附议一次（需防刷防重）。

4.  **提议管理：修改与删除 (Update / Delete)**
    *   在“我的发布”里，用户允许删除自己发起的提议。
    *   **⚠️ 强业务逻辑防错点（重要）**：允许用户修改提议。但出于对附议者的公平性考虑，**一旦提议的核心内容被修改，原有的投票/附议数必须强制清零归零**。这部分清零逻辑可以由后端在处理 Update 接口时内部执行，并在执行前能给予调用方足够的状态以便弹窗确认。

### 3. 数据隔离与持久化建设
由于“提议”本身代表那些还未被证明存在的“占位课程”，它与现有的“实体课程数据库表”应该存在状态隔离：
*   建议可以新增状态字段（如：`status: pending / approved`）或者干脆使用独立的“提议表”与正式的“课程表”隔离。
*   期待有一套后台管理逻辑：当某个提议的投票数达到阈值，或管理员在后台点击“收录”后，该提议状态转正，顺滑迁移进核心的可用课程库中。
