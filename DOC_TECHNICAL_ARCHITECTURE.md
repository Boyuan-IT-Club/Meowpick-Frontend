# 喵选（Meowpick）前端运行逻辑与技术架构指南

本文档面向后续接手或测试该项目的人员，主要用通俗概念解析当前版本中核心功能的底层实现思路。

## 1. 列表渲染与状态管理
我的发布页面的核心是一个响应式的状态流。前端在内存中维护了一个全量的复合列表数据（包含所有的吐槽和提议），同时记录用户当前选中的过滤标签。
当用户点击切换全部、吐槽或提议时，前端会自动触发过滤机制，并将结果按时间倒序排列渲染到视图上。这种声明式的数据驱动避免了频繁的网络请求和手动页面重绘。

## 2. 路由分发策略
由于搜索结果和我的发布中存在混合类型的数据卡片，前端抛弃了冗长生硬的 if-else 判断跳转。
我们采用了轻量级的策略映射：每张卡片的数据都自带一个 	ype（类型）标识。用户点击卡片时，程序会直接通过 	ype 匹配到对应的路由地址。例如，识别到是吐槽卡片，就自动切入对应的课程评价页；识别到是提议卡片，则进入提议专属详情页。

## 3. 吸顶防穿透技术（负向延伸遮罩法）
为了提升体验，我的页面的横向菜单在页面下滑时会吸附在屏幕上方，且带有毛玻璃半透明效果。但由于原生机制的限制，下方滚动的列表往往会从菜单上方的状态栏区域漏出来。
我们在不增加任何 JS 滚动监听性能损耗的前提下，用纯 CSS 解决了这个问题：在吸附菜单的最上沿，绑定了一个与页面底色完全一致的隐形墙（伪元素），这堵墙自动向上铺满屏幕。当列表元素向上滑动超过菜单时，会直接被这堵墙完美遮挡，实现了原生的沉浸式滚动体验。

## 4. 样式复用与渲染降级
搜索页和个人发布页的卡片 DOM 结构做到了最大程度的统一。为了在不同类型的卡片之间做视觉区分，前端并没有引入更重的边框节点或额外的指示条图片，而是纯粹采用 CSS 环境渐变色（如淡粉色到白色的高斯渐变）。这种方式在保证有区分度又不分轻重的视觉要求下，也将浏览器的层级渲染开销压到了最低。
